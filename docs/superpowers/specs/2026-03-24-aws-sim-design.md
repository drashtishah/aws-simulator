# Plan: AWS Incident Simulation Learning System

## Context

The user is preparing for the AWS Solutions Architect exam and wants to learn by doing -- not MCQs. The idea: realistic incident simulations where you play the Incident Commander, diagnose what broke in a fictional company's AWS infrastructure, and fix it through conversational investigation. Two Claude Code skills (`create-sim` and `play`) power the system, with agent teams providing a live multi-agent simulation at runtime.

This replaces the previous Strands SDK / React / Bedrock deployment approach with a pure Claude Code skill-based system that runs locally.

## Sim Package Structure

Each sim is a directory under `sims/` containing:

### manifest.json

Machine-readable definition with these sections:

- **Metadata**: id, title, difficulty (1-4), category, services, tags, estimated_minutes, prerequisites, exam_topics
- **Company**: name, industry, size -- makes scenarios feel real
- **Team**: narrator config (personality, story_beats, hints, max_hints_before_nudge) + agents array (name, service, artifacts list, capabilities list)
- **Resolution**: root_cause string, fix_criteria array (id, description, required boolean), learning_objectives array

Difficulty levels:
- 1 = Foundational (EC2, S3, IAM, VPC basics)
- 2 = Associate (security, resilience, performance, cost)
- 3 = Developer/Security specialty (Lambda, API Gateway, GuardDuty, WAF)
- 4 = Professional (multi-account, migration, disaster recovery, Transit Gateway)

### story.md

Obsidian-formatted narrative with frontmatter tags. Sections: Opening (read by Narrator at start), plus any mid-scenario story beats, plus Resolution (read when fix is validated).

### resolution.md

Human-readable explanation of root cause, the correct fix, why it works, and what to learn. Links to AWS documentation.

### artifacts/architecture.txt

Every sim MUST include an ASCII architecture diagram as `artifacts/architecture.txt`. This is presented to the user immediately after the opening narrative, before they start investigating. Gives the user a visual mental model of the infrastructure they are debugging.

### artifacts/ (other)

Native-format files that service agents serve on demand:
- `.json` -- IAM policies, bucket policies, CloudTrail events, security group rules, CloudFormation templates
- `.txt` -- CloudWatch logs, access logs, application logs
- `.csv` -- Metrics data (timestamps, CPU, memory, error counts, request rates)
- `.mermaid` -- Architecture diagrams
- Other formats as needed per scenario

## AWS Services Catalog

**Location:** `services/catalog.csv`

A comprehensive CSV file listing every AWS service relevant to certification exams. Both skills use it:
- **create-sim reads it** to identify services with low user knowledge and prioritize sim generation for those gaps
- **play writes to it** after each session, updating knowledge scores based on what the user demonstrated

### CSV Structure

```csv

```

### Fields

- **service**: Short identifier (used in manifest.json service references)
- **full_name**: Official AWS service name
- **category**: compute, storage, database, networking, security, integration, management, analytics, migration
- **exam_priority**: 1 (always tested) through 4 (rarely tested)
- **difficulty_level**: 1-4, matching sim difficulty levels
- **cert_relevance**: Semicolon-separated cert codes where this service appears
- **knowledge_score**: 0-5, updated by play skill (0=never seen, 1=encountered, 2=understands basics, 3=can diagnose issues, 4=can architect with it, 5=mastered)
- **sims_completed**: Count of sims involving this service that the user completed
- **last_practiced**: ISO date of last sim involving this service
- **notes**: Free-text notes from play skill's coaching observations

### services/catalog.md

Generated from the CSV for Obsidian browsing. Grouped by category, sorted by exam_priority. Shows knowledge scores as a visual progress indicator. Regenerated by play skill after updates.

### How create-sim uses it

When generating new sims, create-sim:
1. Reads catalog.csv
2. Identifies services with knowledge_score < 2 and high exam_priority
3. Prioritizes scenario topics that involve those services
4. Ensures new batch covers services the user has not yet practiced

### How play uses it

After each completed sim, play:
1. Reads catalog.csv
2. For each service involved in the sim, updates knowledge_score based on:
   - Did the user ask relevant questions about this service? (+1)
   - Did the user correctly diagnose an issue in this service? (+1)
   - Did the user demonstrate understanding of the service's configuration? (+1)
3. Updates sims_completed count
4. Updates last_practiced date
5. Adds coaching notes if relevant
6. Regenerates catalog.md

## Skill: create-sim

**Location:** `.claude/skills/create-sim/SKILL.md`

### Workflow

1. User invokes `create-sim` (optionally with topic area: "security", "compute", "networking", etc.)
2. Reads `services/catalog.csv` to identify knowledge gaps (low knowledge_score + high exam_priority)
3. Skill uses `WebSearch` to find realistic AWS incident patterns, postmortems, common misconfigurations -- targeting gap services
4. Cross-references findings against `references/exam-topics.md` to ensure exam coverage
5. Proposes 5-6 scenario topics: title, target services, difficulty, 1-line pitch, which knowledge gaps they address
6. User approves/rejects/modifies the topic list
6. For each approved topic, generates the full sim package:
   - Reads `references/sim-template.md` for structure guidance
   - Validates manifest against `assets/manifest-schema.json`
   - Creates company with realistic name, industry, size
   - Generates all artifacts in native formats
   - Writes story.md with Obsidian frontmatter
7. Registers each new sim in `sims/registry.json`
8. Regenerates `sims/index.md`

### Bundled Resources

- `references/exam-topics.md` -- AWS SA Associate + Professional + Developer + Security exam domains, organized by difficulty level, with key services per domain
- `references/sim-template.md` -- Complete annotated example of one sim package (manifest + story + resolution + artifacts) showing the expected format and quality bar
- `assets/manifest-schema.json` -- JSON schema for manifest.json validation

## Skill: play

**Location:** `.claude/skills/play/SKILL.md`

### Workflow

1. User invokes `play`
2. Reads `learning/profile.json` (creates default if first run)
3. Filters `sims/registry.json`: eligible sims = difficulty <= current_level AND not in completed_sims AND prerequisites met
4. Presents available sims with title, difficulty, category, estimated time
5. User picks a sim
6. Reads the sim's `manifest.json`
7. Checks for existing session state file at `learning/sessions/{sim-id}.json`:
   - If found: resumes sim from saved state (criteria already met, hints used, elapsed context)
   - If not found: starts fresh
8. Creates an agent team via `TeamCreate`:
   - Narrator agent (team lead): full story, resolution criteria, hint list, story beats, resume state if any
   - N service agents (teammates): each gets their artifact files + capability-scoped system prompt
9. Narrator delivers the Opening from story.md
10. Narrator immediately presents the ASCII architecture diagram from `artifacts/architecture.txt`
11. User investigates conversationally:
    - Asks questions -> routed to appropriate service agent
    - Service agents respond with data from their artifacts only (no interpretation)
    - Narrator delivers time-based story beats for urgency
    - Narrator offers hints progressively after failed investigation paths
12. User proposes a fix -> Narrator validates against fix_criteria
13. On resolution:
    - Narrator reads the Resolution section of story.md
    - Narrator delivers learning summary with objectives from manifest
    - Play skill updates learning/profile.json
    - Play skill updates services/catalog.csv with knowledge scores
    - Play skill appends session to learning/journal.md
    - Play skill provides coaching notes based on investigation patterns
    - Play skill deletes the session state file (sim is complete)
    - Play skill commits all changes (profile.json, catalog.csv, journal.md) and pushes to remote
    - Play skill tells user: "Sim complete. Start a new session to play the next one."

### One Sim Per Session

Each play session runs exactly one simulation. After resolution (or if the user quits), the session ends. The user starts a new Claude Code session to play another sim. This keeps context windows clean and prevents token exhaustion from multi-sim sessions.

### Session Persistence

If a user needs to stop mid-sim (closes terminal, runs out of time), play saves state to `learning/sessions/{sim-id}.json`:

```json
{
  "sim_id": "001-s3-bucket-breach",
  "started_at": "2026-03-24T14:30:00Z",
  "last_active": "2026-03-24T14:45:00Z",
  "criteria_met": ["identify_public_policy"],
  "criteria_remaining": ["identify_when_changed", "propose_fix"],
  "hints_used": 1,
  "questions_asked": 8,
  "investigation_summary": "User identified the bucket policy issue but has not yet checked CloudTrail or proposed a fix.",
  "status": "in_progress"
}
```

**Automatic saving:** The Narrator agent writes to the session state file automatically after every significant interaction:
- After each question the user asks (updates questions_asked count)
- After each hint dispensed (updates hints_used)
- After each fix_criteria is met (updates criteria_met)
- Updates investigation_summary with a rolling 2-3 sentence description of where the user is

This means if the terminal closes unexpectedly, the latest progress is already on disk. No explicit "save" action needed from the user.

When play starts and finds an existing session file for a sim, it presents the option to resume. The Narrator uses the `investigation_summary` to provide context continuity ("Welcome back. When we left off, you had identified the public bucket policy...").

### Learning Tracking

**learning/profile.json:**
```json
{
  "current_level": 1,
  "completed_sims": [],
  "unlocked_levels": [1],
  "service_exposure": {},
  "question_patterns": {},
  "weaknesses": [],
  "strengths": [],
  "total_sessions": 0,
  "last_session": null
}
```

**Progression rules:**
- Level 1 unlocked by default
- Complete 2 sims at level N to unlock level N+1
- Per-service prerequisites: some advanced sims require completing an intro sim for that service first (defined in manifest.prerequisites)

**Coaching:**
- Play skill analyzes question patterns across sessions (what the user checks first, what they miss)
- After each session, provides actionable coaching (e.g., "You jumped straight to the fix without checking CloudTrail -- consider investigating the timeline first next time")
- Updates weaknesses/strengths arrays based on patterns
- When presenting sim options, surfaces sims that target identified weaknesses

### Bundled Resources

- `references/agent-prompts.md` -- System prompt templates for Narrator and service agents, with placeholders for sim-specific data
- `references/coaching-patterns.md` -- Rules for analyzing question patterns and generating coaching feedback

## Agent Team Design

### Team Creation

When play activates a sim, it creates:

```
Team: "sim-{id}"
  Narrator (team lead)
    - System prompt: personality + full story + resolution criteria + hints + story beat triggers
    - Role: drive narrative, validate fixes, deliver coaching, manage pacing

  Service Agent 1..N (teammates)
    - System prompt: "You are the {service} console for {company}. Respond ONLY with data from your loaded artifacts. Do not interpret, analyze, or suggest. Return raw data as it would appear in the AWS console."
    - Each agent loads only its assigned artifact files
    - Capabilities list defines what queries the agent can answer
```

### Message Routing

- User messages go to the team (broadcast or directed)
- Service agents respond when queried about their domain
- Narrator intervenes for story beats, hints, and fix validation
- When user proposes a fix, Narrator checks each fix_criteria and responds with pass/fail

### Session End

Team is cleaned up via `TeamDelete` after:
- Successful resolution (all required fix_criteria met)
- User quits mid-sim (progress noted but sim not marked complete)
- Timeout (optional, if estimated_minutes exceeded by 2x)

## AWS Topic Coverage

Sims should cover all major exam domains across cert levels:

**Level 1 - Foundational:**
- EC2 instance types and states, S3 bucket basics, IAM users/roles/policies, VPC subnets and routing, RDS basics, CloudWatch fundamentals

**Level 2 - Associate (SAA-C03):**
- Security: bucket policies, security groups vs NACLs, KMS encryption, Secrets Manager
- Resilience: Auto Scaling, ELB, RDS Multi-AZ, DynamoDB
- Performance: CloudFront, ElastiCache, EBS optimization, Lambda
- Cost: Reserved vs On-Demand vs Spot, S3 storage classes, right-sizing

**Level 3 - Developer/Security:**
- Lambda cold starts, API Gateway, SQS/SNS patterns, CodePipeline
- GuardDuty, WAF, Shield, Security Hub, Macie, Cognito

**Level 4 - Professional (SAP-C02):**
- Direct Connect, Transit Gateway, multi-account (Organizations, SCPs)
- Database migration (DMS), disaster recovery (RPO/RTO)
- Hybrid architectures, containerization (ECS/EKS)
