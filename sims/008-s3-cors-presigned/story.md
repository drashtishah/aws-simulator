# The Upload That Worked Everywhere Else

## Opening

It is 2:15 PM on a Thursday. The office is quiet. Someone is microwaving soup in the kitchen. The Slack notification arrives in the #frontend channel: "Upload is broken in the app. Tested the same URL in Postman and it works fine."

Pollen Systems builds a document management platform for small law firms. Seventeen engineers. Series A. The product lets paralegals upload case files, contracts, and scanned depositions. The upload feature is the core of the product. It has worked for eight months without incident.

The frontend developer, Rina, added a new upload component last week. It uses presigned URLs generated by the backend through API Gateway. The backend calls S3's `createPresignedUrl` for PUT operations and returns the URL to the browser. The browser then uploads the file directly to S3 through CloudFront.

Rina tested the presigned URL with curl. It worked. She tested it with Postman. It worked. She tested it with a Python script. It worked. She opened the browser, clicked Upload, and watched the request fail. No response body. No useful error message in the application. Just a red line in the Network tab.

She checked the IAM policy attached to the role that generates the presigned URLs. The permissions were correct. She checked the presigned URL expiry. It was set to 3600 seconds. She regenerated the URL. Same failure. She read the IAM policy again, slowly. She read it a third time. Two hours passed.

The bucket is called `pollen-docs-prod`. The CloudFront distribution sits in front of it. The API Gateway endpoint at `api.pollen.io` generates the presigned URLs. The application frontend runs at `app.pollen.io`. These are different origins.

A backend engineer, Tomás, joined the thread at 4:10 PM. He checked the API Gateway CloudWatch logs. The presigned URL generation was succeeding every time. He checked the S3 access logs. Uploads from the backend test script were completing. He asked Rina if she had looked at the browser console. She had not. She opened it.

The console showed red text. It said something about CORS. It said the response to the preflight request did not have an acceptable `Access-Control-Allow-Origin` header. Rina had not encountered CORS before. The error did not appear in curl. It did not appear in Postman. It only appeared in the browser.

The S3 bucket has no CORS configuration. It has never needed one, because until now, all uploads went through the backend. The CloudFront distribution is caching responses without considering the `Origin` header. These are two separate problems that produce the same symptom.

## Resolution

The issue had two parts. The S3 bucket `pollen-docs-prod` had no CORS configuration at all. When the browser sent a preflight OPTIONS request, S3 returned a 403 with no CORS headers. The browser refused to proceed with the PUT.

The second part was CloudFront. The distribution's cache behavior used the default cache policy, which did not include `Origin`, `Access-Control-Request-Headers`, or `Access-Control-Request-Method` in the forwarded headers. Even after adding CORS configuration to S3, CloudFront would cache the first response -- which might not include CORS headers if the first request came from a non-browser client -- and serve that cached response to all subsequent requests regardless of their `Origin` header.

The fix was two changes. First, a CORS configuration was added to the S3 bucket allowing PUT and POST methods from `https://app.pollen.io`. Second, the CloudFront distribution's cache behavior was updated to use an origin request policy that forwards the three CORS-related headers to S3. A CloudFront cache invalidation was created for `/*` to clear stale responses.

Rina's upload worked on the first try after both changes were applied. The entire investigation had taken two hours and forty minutes. Two hours of that was spent reading IAM policies. The browser console had the answer from the beginning.

Tomás added a comment to the team wiki: "If the upload works in curl but not in the browser, check CORS first." It was a short sentence. It would save someone two hours someday.
