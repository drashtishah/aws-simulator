# The Upload That Worked Everywhere Else

## Opening

company: Pollen Systems
industry: document management for small law firms, Series A, 17 engineers
product: paralegals upload case files, contracts, scanned depositions -- upload feature is core of product, worked eight months without incident
time: 2:15 PM, Thursday
scene: Slack notification in #frontend channel reporting upload broken in app
alert: "Upload is broken in the app. Tested the same URL in Postman and it works fine."
stakes: upload feature is core product functionality for law firm customers
early_signals:
  - frontend developer Rina added new upload component last week using presigned URLs generated by backend through API Gateway
  - backend calls S3 createPresignedUrl for PUT operations, returns URL to browser, browser uploads directly to S3 through CloudFront
  - presigned URL works with curl, Postman, and Python script -- fails only in the browser
  - no response body, no useful error message in the application, just a red line in Network tab
  - Rina checked IAM policy (correct), presigned URL expiry (3600 seconds), regenerated URL (same failure)
  - Rina read the IAM policy three times over two hours
  - bucket is pollen-docs-prod, CloudFront distribution in front of it, API Gateway at api.pollen.io generates presigned URLs, frontend at app.pollen.io (different origins)
  - backend engineer Tomas joined thread at 4:10 PM, checked API Gateway CloudWatch logs (presigned URL generation succeeding), checked S3 access logs (backend test script uploads completing)
  - Tomas asked if anyone checked the browser console -- Rina had not
  - browser console showed CORS error: response to preflight request did not have acceptable Access-Control-Allow-Origin header
investigation_starting_point: the upload works from curl, Postman, and Python but fails in the browser. The S3 bucket has no CORS configuration (never needed one because all uploads previously went through the backend). The CloudFront distribution caches responses without considering the Origin header. These are two separate problems producing the same symptom.

## Resolution

root_cause: two-part issue. S3 bucket pollen-docs-prod had no CORS configuration -- browser preflight OPTIONS request returned 403 with no CORS headers. CloudFront distribution used default cache policy that did not forward Origin, Access-Control-Request-Headers, or Access-Control-Request-Method headers to S3.
mechanism: without CORS config, S3 returned 403 on OPTIONS preflight and browser refused to proceed with PUT. Even after adding CORS to S3, CloudFront would cache the first response (possibly from a non-browser client without CORS headers) and serve it to all subsequent requests regardless of Origin header.
fix: two changes applied together. First, CORS configuration added to S3 bucket allowing PUT and POST methods from https://app.pollen.io. Second, CloudFront cache behavior updated to use origin request policy forwarding the three CORS-related headers to S3. CloudFront cache invalidation created for /* to clear stale responses. Upload worked on first try after both changes.
contributing_factors:
  - no CORS configuration on S3 bucket because all uploads previously went through the backend
  - CloudFront default cache policy does not forward CORS-related headers
  - two hours spent reading IAM policies before anyone checked the browser console
  - Tomas added team wiki note: "If the upload works in curl but not in the browser, check CORS first."
