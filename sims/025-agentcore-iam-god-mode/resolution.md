# Resolution: The Agent With the Wrong Keys

## Root Cause

The IAM execution role on the `brackish-data-insights-agent` AgentCore Runtime had a wide-open inline policy: `s3:*`, `kms:Decrypt`, and `secretsmanager:GetSecretValue` with `Resource: "*"`. That policy was not written by a human. It was generated by the AgentCore starter toolkit's auto-create-role helper on the very first deploy. The helper is meant for local development; the AWS documentation for it explicitly says it is not recommended for production. No one replaced the role before the agent went live. For five weeks the agent's identity could read every object in every bucket in the AWS account. Its tool-use loop, picking buckets by keyword-matching against the user's question, occasionally chose the payroll bucket and the customer-PII bucket.

## Correct Remediation

1. **Find the caller identity fast.** Filter CloudTrail on the payroll and PII buckets for the last seven days with the `UserIdentity.ARN` equal to the agent's execution role. The resulting list of GetObject events is the scope of the exposure. Save it; you will need it for the incident writeup.
2. **Read the policy, do not trust the mental model.** Run `aws iam get-role-policy` on the execution role's inline policy. If any statement has `Resource: "*"` for `Action: s3:*`, `kms:Decrypt`, or `secretsmanager:GetSecretValue`, you have found the root cause regardless of what the app-level config says.
3. **Stop the bleeding with an explicit `Deny`.** Attach a second inline policy (or a managed policy) to the role that `Deny`s `s3:Get*`, `s3:List*`, and `s3:Put*` on the specific sensitive bucket ARNs. `Deny` always beats `Allow` in IAM evaluation, so this takes effect immediately and does not require any coordination with the team that owns the agent.
4. **Write the real least-privilege policy.** Scope `s3:GetObject` and `s3:ListBucket` to exactly the bucket and prefix the agent is supposed to read: `arn:aws:s3:::brackish-analytics-reports` and `arn:aws:s3:::brackish-analytics-reports/*`. Remove `kms:*` and `secretsmanager:*` unless the agent truly needs them. If it does, scope them to specific key and secret ARNs.
5. **Swap atomically.** Use `aws iam put-role-policy` to replace the old inline policy in a single call, then remove the temporary Deny. Verify the agent can still answer a known-good question before declaring the fix complete.
6. **Add IAM Access Analyzer on the account.** Enable external-access and unused-access findings. Access Analyzer would have surfaced this role as over-permissioned long before the researcher did. Route findings to a Slack channel the security team watches.
7. **Add a resource-level allowlist.** Put a bucket policy (or a Resource Control Policy at the Organizations level) on the payroll and PII buckets that allows only specific identity ARNs, scoped by an `aws:PrincipalOrgID` condition or a specific role list. That way even a future over-permissioned role cannot accidentally read these buckets.
8. **Decide what to notify.** Because the policy technically granted read access to regulated data for five weeks, and the agent actually exercised that access dozens of times, the incident is customer-reportable under the enterprise contracts. Produce a timeline from CloudTrail, identify which customer records were in the read objects, and run the notification through legal.

## Key Concepts

### Why agent execution roles need tighter scoping than typical workloads

A traditional service has a fixed list of API calls it makes. You can read the code and enumerate every AWS action that could possibly be called, then write a policy that allows exactly those actions. An AI agent does not work that way. The agent's tool-use loop is an LLM that decides, at runtime, what to do next based on a natural-language question. Every permission the role holds is reachable by a creative prompt. The right question when writing an agent's policy is not "will the code call this?" but "am I comfortable with this permission being chosen by the planner on any given turn?" Wildcards fail that test by definition.

### The AgentCore starter toolkit's auto-create-role helper

The AgentCore starter toolkit ships a convenience helper that creates an execution role for you when you are trying to spin up an agent in a few minutes. The role it creates grants broad permissions because, in development, you usually do not know yet which services the agent will need to call. The AWS documentation for this helper explicitly says it is for development and testing, not for production. The warning is easy to miss because the helper just works and the agent runs fine. Replacing it with a scoped role is a deliberate step that has to happen before the agent serves real users. In practice, many teams forget; this incident is a direct consequence of that gap.

### Defense in depth with a data perimeter

Identity-based least privilege ("this role can only do X") is one line of defense. It fails when a role is accidentally over-scoped. A data perimeter is the second line: a resource-level rule that says "this bucket only accepts access from these specific principals." It does not matter what a new role's policy allows; if the bucket policy does not explicitly permit that principal, the request is denied. For regulated data, the two together are the expected baseline: scoped identity policies everywhere, plus explicit allowlists on the sensitive resources.

## Other Ways This Could Break

### The execution role uses the AWS-managed `AdministratorAccess` policy
Instead of an overly broad inline policy, the role has `AdministratorAccess` attached. The effect is the same (the agent can do anything the account can do), but because `AdministratorAccess` is a named managed policy it is harder to gloss over in a review.
**Prevention:** Forbid attaching `AdministratorAccess` to any agent role via a Service Control Policy. Require agent roles to use inline or customer-managed policies that a human has reviewed. IAM Access Analyzer's external-access findings will flag administrator-level roles specifically.

### The role is scoped correctly, but the bucket policy grants any role in the account read access
The identity-based policy on the role is fine. The failure is on the resource: a bucket policy with `Principal: {"AWS": "arn:aws:iam::ACCOUNT:root"}` and `s3:GetObject` allows any role in the account to read the bucket. Any new role, agent or otherwise, inherits read access just by existing.
**Prevention:** Do not use root-account principals on sensitive bucket policies. List specific role ARNs, or use `aws:PrincipalOrgID` and `aws:PrincipalTag` conditions to restrict access to a specific Organizational Unit or workload label.

### The agent's role is scoped, but a tool handler assumes a more privileged role to do its work
The AgentCore execution role is perfect on paper. But one of the tools the agent can invoke is a Lambda function that calls `sts:AssumeRole` to obtain credentials for a higher-privileged role before reading S3. From the outside the agent looks safe; in reality its effective access is whatever the secondary role allows.
**Prevention:** Treat every transitive role chain as part of the agent's effective policy. Audit tool-handler code for `AssumeRole` calls. Prefer passing specifically scoped temporary credentials from the agent directly, so the chain is short and visible.

## SOP Best Practices

- Never ship an agent to production with a starter-toolkit-generated execution role. Treat replacing the auto-generated role with a scoped one as a required step in the agent's infrastructure-as-code template.
- Enable IAM Access Analyzer on every account that hosts AgentCore agents. Use both external-access findings and unused-access findings. Review weekly.
- Build a data perimeter with Service Control Policies and Resource Control Policies so that sensitive buckets have a resource-level allowlist. Identity least privilege alone is not a complete control for data classified as restricted.
- Write agent IAM policies as if the agent's tool loop is adversarial. Every wildcard is a hypothetical prompt injection waiting to happen. Prefer many narrow permissions over one broad one.

## Learning Objectives

1. **Agent identity as attack surface**: Understand why AI agents require tighter IAM scoping than typical workloads, because the agent chooses its own API calls at runtime.
2. **CloudTrail forensics by principal**: Use CloudTrail to reconstruct every API call made by a specific IAM role over a time window.
3. **Starter-toolkit dangers**: Recognize the AgentCore starter toolkit's auto-create-role helper and understand why it is development-only.
4. **Data perimeter controls**: Combine identity least privilege with Resource Control Policies and bucket-level allowlists so no single over-permissioned role can expose regulated data.
