# The Agent With the Wrong Keys

## Opening

- company: Brackish Data
- industry: B2B data enrichment analytics
- product: a natural-language analytics assistant that answers customer-segmentation and campaign-performance questions by reading reports from a curated analytics data lake
- scale: thirty engineers, about two hundred B2B customers on the self-serve plan, forty on an enterprise plan with a data-processing addendum that specifically prohibits cross-bucket reads
- time: Wednesday 11:47 AM local
- scene: head of security is at a partner offsite, CTO is in a board meeting. The engineering lead for the insights-agent team is the person who picks up the page.
- alert: An external security researcher emails: "Your AI assistant is answering questions with content that appears to be from internal payroll records. Screenshots attached."
- stakes: the forty enterprise customers have contracts that explicitly forbid internal cross-dataset reads by shared systems. If the agent has been touching payroll data for any of the five weeks it has been live, this is a breach notification under SOC 2 and under the customer contracts.
- early_signals:
  - The external researcher's screenshots show the agent paraphrasing a payroll CSV header row and a named manager's salary band in a marketing-strategy answer
  - A follow-up question asked by the engineering lead confirms the agent can summarize rows from a customer-PII bucket
  - CloudTrail data events on the payroll and PII buckets show hundreds of GetObject calls over the last twelve hours, all with the same role principal
  - The agent has been running on AgentCore for five weeks; nobody reviewed the execution role after the initial "ship it" commit
- investigation_starting_point: the engineering lead has CloudTrail open, showing repeated GetObject calls on buckets the agent was not supposed to read. The Principal is always the same: arn:aws:iam::498231746283:role/brackish-data-insights-agent-role. The team's mental model was that the agent could "only read the analytics bucket." Something in that model is wrong.

## Resolution

- root_cause: the AgentCore execution role brackish-data-insights-agent-role was not written by hand. It was generated by the AgentCore starter toolkit's auto-create-role helper during the first deploy. The helper attaches an inline policy that grants s3:*, kms:Decrypt, and secretsmanager:GetSecretValue with Resource: "*". The intent is to get new agents running quickly in development; the docs warn against using this role in production. Nobody replaced it before the agent went live. For five weeks the agent had full read access to every bucket in the account, and its tool-use loop occasionally chose to read from the payroll and customer-PII buckets when a question was phrased ambiguously ("what does the team think about retention?" led the agent to skim the HR-review bucket; "who is our biggest account?" led it to read customer-PII).
- mechanism: the agent's planner is a tool-using LLM. Given a question, it first lists the buckets it can see via s3:ListAllMyBuckets, then picks a candidate bucket from the list based on the question. Because Resource: "*" allows ListAllMyBuckets, the planner sees all twenty-seven buckets in the account, not just the analytics one. Some of those bucket names are suggestive (brackish-payroll-exports, brackish-customer-pii), and the planner correlates keywords in the user's question with bucket names. On a well-scoped role it would have seen only the analytics bucket and had no choice.
- fix: the engineering lead first attaches an explicit Deny policy to the role covering the payroll and PII bucket ARNs. They then write a new inline policy scoped to arn:aws:s3:::brackish-analytics-reports and arn:aws:s3:::brackish-analytics-reports/*, strip out kms:* and secretsmanager:*, and swap the role's inline policy in one put-role-policy call. The agent is redeployed with the same role but the new policy; no code changes are required. Follow-ups include an Access Analyzer setup, a Resource Control Policy on the two sensitive buckets, and an incident notification to the forty enterprise customers whose contracts require it.
- contributing_factors:
  - The AgentCore starter toolkit's auto-create-role default grants account-wide wildcards. The docs warn this is development-only but the warning is easy to miss.
  - The team's mental model of the agent ("it can only read the analytics bucket") was based on application-level config, not on the IAM policy. Nobody read the actual policy document before ship.
  - There was no IAM Access Analyzer set up on the account, so the role's unused permissions were never flagged.
  - The payroll and PII buckets had no resource-level allowlist. Their bucket policies did not explicitly Deny other internal roles, so identity-level least privilege was the only line of defense.
  - S3 data events on the sensitive buckets were enabled (this is how the investigation succeeded), but no alarm fired on unusual caller identities.
