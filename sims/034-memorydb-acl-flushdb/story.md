# The Key the Cache Would Not Drop

## Opening

- company: Drifter Analytics
- industry: Mobile game analytics SaaS
- product: real-time analytics dashboard for mobile game studios; tracks per-player, per-event telemetry and renders studio-side funnels and retention
- scale: Series A, 31 engineers, 240 game studios as customers, ~18,000 dashboard logins per day, MemoryDB cluster sized at 2 shards x 3 nodes (cache.r6g.large)
- time: Monday 10:14 Eastern, one week after the weekend migration from Amazon ElastiCache to Amazon MemoryDB
- scene: Customer Support Slack channel. 22 tickets open. Two studio accounts escalated to the CS Director. QA engineers at those studios cannot stay logged in long enough to run test batches.
- alert: no PagerDuty alert fired. CloudWatch shows DatabaseMemoryUsagePercentage climbing for six days but the alarm threshold was set at 95 percent with a 60-minute evaluation window, and the metric is currently at 94.
- stakes: Two enterprise accounts representing 38 percent of ARR are escalated. Drifter's QBR with one of them is Thursday. Every hour without a fix is another wave of invalid-session tickets. Memory is at 94 percent and eviction has started, meaning real session state is being lost.
- early_signals: 22 customer support tickets citing 'invalid session' or 'got logged out', CloudWatch showing MemoryDB memory climbing toward 95 percent and Evictions non-zero, no obvious deploy or traffic change
- investigation_starting_point: You know the MemoryDB migration happened last weekend. You have access to the MemoryDB console, ECS task logs, CloudWatch metrics, and the Drifter code repository for the cleanup-scratch task.

## Resolution

- root_cause: The Redis ACL rule attached to the drifter-app user on the MemoryDB cluster is 'on ~* +@read +@write +@connect -@dangerous'. The last fragment, -@dangerous, strips out every command in Redis's @dangerous category, which includes FLUSHDB, FLUSHALL, SHUTDOWN, CONFIG, KEYS, and DEBUG. The nightly cleanup-scratch ECS task authenticates as drifter-app (like every other workload does), calls FLUSHDB to empty the scratch pre-aggregation database (db 5), and receives '(error) NOPERM this user has no permissions to run the FLUSHDB command'. The task retries three times, fails, exits non-zero, and logs a single-line failure that nobody alarms on.
- mechanism: The pre-aggregation service continues writing to db 5 all day every day. The nightly cleanup is the only thing that ever removed those keys. Under ElastiCache the cleanup worked for 18 months because ElastiCache in open-access mode does not check ACLs. MemoryDB always checks ACLs, and the migration wizard auto-generated the drifter-app user with the 'safe' default rule excluding @dangerous. Scratch keys therefore accumulate at ~300 MB/day. By day 6 they have consumed 2.1 GB of the 3.2 GB shard capacity. MemoryDB's maxmemory-policy is the default allkeys-lru, so once DatabaseMemoryUsagePercentage crosses 95 percent, real session-state keys start getting evicted. Players hit 'invalid session' on the next API call after their session token was evicted.
- fix: Create a dedicated MemoryDB user drifter-cleanup with rules 'on ~* +ping +select +flushdb'. Attach it to the existing cluster ACL via UpdateACL. Update the cleanup-scratch ECS task definition to reference a Secrets Manager secret for the new user's credentials. After the deploy, run the cleanup manually: db 5 drops from 2.1 GB to 48 MB within seconds. DatabaseMemoryUsagePercentage falls to 62 percent. Evictions return to zero over the next 20 minutes as writes repopulate what was lost. Session-login failures stop. Customer Support confirms ticket influx halts at 11:02.
- contributing_factors: The migration from ElastiCache to MemoryDB was done as a Saturday overnight cutover with a test plan that covered session reads, session writes, cluster failover, and snapshot restore; the nightly cleanup task was not included in the test plan because 'it just writes to Redis, same as the app'. The ACL was generated by the MemoryDB console's Create Cluster wizard and accepted as-is; nobody opened the rule string or cross-referenced the commands it blocked against the list of commands the workloads use. The cleanup-scratch ECS scheduled task had no alarm on non-zero exit code. MemoryDB's DatabaseMemoryUsagePercentage alarm was set at 95 percent with a 60-minute evaluation, so a six-day slow climb never crossed it. A shared user (drifter-app) authenticated every workload rather than one user per operational role.
